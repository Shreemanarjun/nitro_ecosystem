# Lifecycle and Resource Management

This document explains how Nitrogen manages the lifetime of native resources,
why apps can crash after a hot restart, and the patterns — `NativeFinalizer` and
`NativeCallable.listener` — that prevent those crashes.

---

## The dangling-reference crash

When Flutter performs a **hot restart**:

1. The Dart VM tears down the current isolate — all Dart objects are abandoned.
2. `ReceivePort`s are invalidated; `StreamSubscription.cancel()` is **not** called.
3. But native threads keep running — a Kotlin `Flow` coroutine or Swift Combine
   pipeline is still alive and still trying to emit.
4. When native code calls `Dart_PostCObject(port, value)` on the now-dead port,
   the result is undefined — at best a no-op, at worst a segfault.

The same crash happens when a widget is disposed without cancelling its stream
subscription: the native emitter runs forever, consuming memory and CPU, and
eventually posts to a GC'd `ReceivePort`.

---

## How Nitrogen's runtime addresses this

### Scenario 1 — Explicit cancel (the happy path)

```dart
final sub = MySensor.instance.readings.listen((r) { ... });

// Later:
sub.cancel(); // ← triggers onCancel → release(port) → ReceivePort.close()
```

The generated `_MySensorImpl` calls `NitroRuntime.openStream(release: ...)`. The
`onCancel` hook calls `release(nativePort)` synchronously. The native `Flow`/
`Publisher` receives the cancellation signal and stops emitting.

**Always cancel subscriptions in `dispose()`:**

```dart
@override
void dispose() {
  _sub?.cancel();
  super.dispose();
}
```

### Scenario 2 — GC without cancel (the safety net)

`NitroRuntime.openStream` attaches a Dart `Finalizer` to every `StreamController`
it creates:

```dart
static final _streamFinalizer =
    Finalizer<void Function()>((doRelease) => doRelease());

// Inside openStream:
_streamFinalizer.attach(controller, doRelease, detach: controller);
```

If `cancel()` is never called and the `StreamController` becomes unreachable, the
GC eventually calls `doRelease()`, which calls `release(nativePort)` and closes
the `ReceivePort`. The native emitter stops.

The `doRelease` closure captures only `release` (a function pointer) and
`nativePort` (an `int`). It holds **no** strong reference back to the controller,
so it does not prevent the controller from being collected.

### Scenario 3 — Hot restart (the hard case)

A Dart `Finalizer` runs Dart code — it requires the Dart event loop to be running.
During a complete isolate shutdown (hot restart), finalizers **may not fire**.

However, `Dart_PostCObject` on an invalidated port returns `false`. The C++ bridge
template checks this return value and stops emitting:

```cpp
// Generated by Nitrogen — inside the emit_ JNI function:
bool posted = Dart_PostCObject_DL(port, &dart_value);
if (!posted) {
  // Port is dead — stop the emitter on the native side.
  // The Flow/Publisher should be cancelled here.
}
```

For **guaranteed** cleanup across hot restarts, use `NativeFinalizer` (see below).

## Pattern 1 — NativeFinalizer

`NativeFinalizer` differs from Dart's `Finalizer` in one critical way: it calls a
**C function directly**, without needing the Dart event loop. It fires even when
the Dart isolate is being torn down.

Use it whenever a Dart object holds a reference to native memory or a native
handle that must be freed.

### How it works

```
Dart object (e.g. ZeroCopyBuffer)
       │
       │ NativeFinalizer.attach(dartObject, nativeToken)
       ▼
   GC detects object is unreachable
       │
       ▼
   C function called: void releaseHandle(void* token)
       │
       ▼
   Native memory/handle freed — even during hot restart
```

### Implementation pattern

**Step 1 — Expose a C release symbol in your native code**

```cpp
// In your plugin's C++ bridge (e.g., my_sensor_release.cpp)
extern "C" {
  // Called by NativeFinalizer when the Dart object is GC'd.
  // token is whatever Pointer<Void> you passed to attach().
  void nitro_release_sensor_handle(void* token) {
    auto* handle = reinterpret_cast<SensorHandle*>(token);
    delete handle;
  }
}
```

**Step 2 — Look up the symbol and create the NativeFinalizer**

```dart
// In your plugin's Dart code (e.g., my_sensor.native.dart or runtime helper)
final _lib = NitroRuntime.loadLib('my_sensor');

final _sensorFinalizer = NativeFinalizer(
  _lib.lookup<NativeFunction<Void Function(Pointer<Void>)>>(
    'nitro_release_sensor_handle',
  ),
);
```

**Step 3 — Attach to every Dart object that wraps a native handle**

```dart
class SensorHandle {
  final Pointer<Void> _nativePtr;

  SensorHandle(this._nativePtr) {
    // When this SensorHandle is GC'd, nitro_release_sensor_handle(_nativePtr)
    // will be called — even during hot restart.
    _sensorFinalizer.attach(this, _nativePtr, detach: this);
  }

  void release() {
    _sensorFinalizer.detach(this);               // cancel the finalizer
    _lib.lookupFunction<Void Function(Pointer<Void>), void Function(Pointer<Void>)>(
      'nitro_release_sensor_handle',
    )(_nativePtr);                               // release immediately
  }
}
```

### NativeFinalizer vs Dart Finalizer

| | `Finalizer<T>` | `NativeFinalizer` |
|---|---|---|
| Callback type | Dart closure | C function pointer |
| Runs during isolate shutdown | No | **Yes** |
| Safe for hot restart | No | **Yes** |
| Requires a native symbol | No | **Yes** |
| Overhead | Very low | Very low |

Use `NativeFinalizer` for any object that wraps a native resource that must be
freed even if the Dart isolate is killed.

Use `Finalizer` for cleanup that is desirable but not safety-critical (e.g.,
cache invalidation).

## Pattern 2 — NativeCallable.listener

A `NativeCallable` turns a Dart function into a C function pointer that native
code can call. There are two variants:

| | `NativeCallable.isolateLocal` | `NativeCallable.listener` |
|---|---|---|
| Callable from | Dart isolate thread only | **Any native thread** |
| Execution model | Synchronous — blocks caller | Posts to Dart event loop |
| Hot restart behaviour | Crashes if called after isolate torn down | Posts silently dropped |
| Use case | Called from within same isolate | Called from Kotlin coroutine / Swift DispatchQueue |

**Nitrogen uses `NativeCallable.listener` for all native-thread callbacks.**

### Why this matters

Kotlin `Flow` collectors run on a `CoroutineDispatcher` thread (not the Flutter
isolate thread). Swift Combine subscriptions run on a `DispatchQueue`. When
native code holds a C function pointer and calls it, it is always from a non-Dart
thread.

Calling an `isolateLocal` callable from a non-Dart thread is **undefined behaviour**
and will crash. `listener` callables are designed for exactly this case.

### Correct pattern for a native completion callback

```dart
// Dart side: create a listener callable
late NativeCallable<Void Function(Int64 result)> _onComplete;

void _startNativeWork() {
  _onComplete = NativeCallable<Void Function(Int64 result)>.listener(
    _handleResult,
  );

  // Pass the function pointer to native code.
  // Native code will call it from a background thread when done.
  _startWorkPtr(_onComplete.nativeFunction);
}

void _handleResult(int result) {
  // This runs on the Dart event loop — safe to update UI.
  print('Native result: $result');
  _onComplete.close(); // release when done
}
```

### Releasing a NativeCallable from native code

A `NativeCallable` **must be closed from the Dart side**, not from native code.

If native code needs to signal that the callable should be released, it must
post a message back to Dart (e.g., via `Dart_PostCObject` on a `ReceivePort`),
and Dart closes the callable in response:

```dart
// Pattern: native → Dart via ReceivePort → Dart closes the callable
final _controlPort = ReceivePort();

void _startNativeWork() {
  _onComplete = NativeCallable<Void Function(Int64)>.listener(_handleResult);

  // Tell native code: here is the callback and here is the control port.
  // When native is done, it posts to controlPort; Dart then closes _onComplete.
  _startWorkPtr(
    _onComplete.nativeFunction,
    _controlPort.sendPort.nativePort,
  );

  _controlPort.listen((_) {
    _onComplete.close();
    _controlPort.close();
  });
}
```

This is the pattern because `NativeCallable.close()` must be called on the
Dart isolate thread, and `ReceivePort` messages always arrive on the Dart
event loop.

## Summary for plugin authors

| Situation | Pattern |
|---|---|
| Native memory that must be freed when Dart object is GC'd | `NativeFinalizer` with a C release symbol |
| Stream that may be abandoned without `cancel()` | `Finalizer` on `StreamController` (Nitrogen does this for you) |
| Native code calls a Dart callback from a background thread | `NativeCallable.listener` (never `.isolateLocal`) |
| Native code wants to signal that a callable can be released | Post to a `ReceivePort`; Dart calls `callable.close()` |
| Stream cleanup on hot restart | `Dart_PostCObject` returns `false` → stop emitting on native side |

---

## What Nitrogen does automatically

| Mechanism | Where | What it prevents |
|---|---|---|
| `Finalizer` on `StreamController` | `NitroRuntime.openStream` | Dangling port reference when subscription is abandoned |
| `Dart_PostCObject` return-value check | Generated C++ bridge | Native emitter keeps running after hot restart |
| `NativeFinalizer` on `ZeroCopyBuffer` | `ffi_utils.dart` | Native buffer leaked when Dart buffer object is GC'd |
| `NativeCallable.listener` | (not yet used — see below) | Crash when native thread calls a Dart callback |

> **Note:** Nitrogen currently uses `ReceivePort` + `Dart_PostCObject` for all
> native-to-Dart delivery, which is the correct and thread-safe mechanism for
> streams. `NativeCallable` would be used for one-shot completion callbacks
> (e.g., when a single async result needs to be delivered from a native thread
> without a port). If you add such a callback in your plugin, use
> `NativeCallable.listener`.
